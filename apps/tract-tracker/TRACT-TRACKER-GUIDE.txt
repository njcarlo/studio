=========================================================


WHAT IS THIS?
-------------
Tract Tracker is a web/mobile app (React Native / Expo) that lets
participants of NTGD share photos of their tract distributions. There is a live
NewsFeed that acts like a public slideshow, and a Correspondent/Photos tab where
users can upload pictures.


WHO CAN DO WHAT
---------------
There are three tiers of users. All of them log in through the app's custom auth
system (not Supabase Auth — a custom tract_users table in Postgres).

  REGULAR USER
  - Gets 1 upload slot total (first come, first served from a global pool of 500)
  - Photo is compressed to 60% quality, forced into a 4:3 crop
  - Once 500 regular-user photos are in the system, the pool is "full" and no
    more regular users can upload

  CORRESPONDENT
  - Unlimited uploads, no slot pool restriction
  - Full resolution (quality 1.0), free crop (no forced aspect ratio)
  - Stored in a separate Google Drive folder from regular users
  - Anyone whose account has is_correspondent = true in the database

  ADMIN
  - Same as Correspondent plus the ability to delete any post in the NewsFeed
  - Currently: njcarlo@gmail.com and cogtv@gmail.com are hardcoded as admins
    even without a database flag (see AuthContext.tsx → ADMIN_EMAILS)
  - isAdmin also implies isCorrespondent


HOW SIGN-IN WORKS
-----------------
Users tap "Sign In" and enter their name + a 6-digit PIN. The app checks the
tract_users table. If the row exists and the PIN matches, the session is stored
in AsyncStorage. There is no email verification or OAuth.

The AuthContext (src/context/AuthContext.tsx) exposes:
  user            — the logged-in user row (or null)
  isAdmin         — true if user.is_admin OR email is in ADMIN_EMAILS
  isCorrespondent — true if user.is_correspondent OR isAdmin
  isTester        — internal flag for test accounts
  signOut()       — clears AsyncStorage session


HOW UPLOADS WORK
----------------
The Photos tab (CorrespondentScreen.tsx) handles all uploads.

Step 1 — Camera or Gallery picker opens via expo-image-picker.
         Quality and crop settings differ by tier (see WHO CAN DO WHAT above).

Step 2 — User fills in a caption. The app automatically appends these hashtags
         to every post before saving, no matter what the user writes:
           #NationalTractsGivingDay
           #OutsideIsBeautiful
           #Connect2Souls
           #BornAgainPilipinas

Step 3 — Slot check (regular users only). The app counts rows in
         correspondent_posts where from_correspondent = false. If that count
         is >= 500, upload is blocked before anything is uploaded.

Step 4 — Image is uploaded to Supabase Storage bucket "correspondent-photos".
         File path: <user_id>/<timestamp>_<username>.jpg

Step 5 — A row is inserted into correspondent_posts with:
           user_id, user_name, image_url, caption (with hashtags), from_correspondent

Step 6 — Fire-and-forget call to the Supabase Edge Function "upload-to-drive".
         This backs the image up to Google Drive. If it fails for any reason
         the app does NOT show an error — the Drive backup is optional insurance,
         not required for the app to work.


GOOGLE DRIVE BACKUP
-------------------
The edge function (supabase/functions/upload-to-drive/index.ts) runs on
Supabase's Deno runtime. It receives the Supabase Storage URL, downloads the
image, and uploads it to Google Drive using a service account.

  Regular user photos  → Drive folder 11mXSMdhs5zR5F_chqCz9N4B11mlb7mL1
  Correspondent photos → Drive folder 11q9nRoxR8XD3EJknur9gLCjbrrN4qb7e

Files are named like:  2026-06-12T14-30-00_JuanDeLaCruz.jpg

FILO MANAGEMENT — If you ever need to free up Drive space:
  1. Open the Drive folder
  2. Sort by Name, descending (newest timestamp = top of list)
  3. Delete from the top — this removes the most recent uploads first,
     preserving the oldest historical records (Last-In First-Out)

Secrets are stored in Supabase (never in the app bundle or git):

Drive API calls are free up to 15 GB of storage per Google account. The app
does not use the paid Drive API — it only uses the free Drive.File scope.


THE NEWS FEED (SLIDESHOW)
-------------------------
NewsFeedScreen.tsx shows all uploaded photos in two ways:

  HERO SLIDESHOW — top section, tall image that auto-advances every 9 seconds.
    - Fades between posts using React Native Animated.
    - Clicking left/right arrows or tapping a filmstrip thumbnail jumps to that post.
    - Resets the 9-second timer whenever you navigate manually.

  CARD FEED — scrollable list below the hero, Facebook-style.
    - Each card shows: user avatar (initials), username, caption, photo, hashtags.
    - On wide screens (>= 1024px / monitor) cards max out at 620px wide.

  ADMIN DELETE — if you are logged in as an admin, a red trash button appears
    on the hero overlay AND on each card in the feed. Tapping it shows a
    confirmation dialog, then:
      1. Deletes the row from correspondent_posts
      2. Deletes the image file from Supabase Storage
      3. Removes the post from the local state (instant UI update)

Layout is fully responsive using useWindowDimensions() — the hero height,
font sizes, and card widths all adjust based on the actual browser/device width.


DATABASE SCHEMA (RELEVANT TABLES)
----------------------------------
  tract_users
    id, name, pin, email, is_correspondent, is_admin, created_at

  correspondent_posts
    id, user_id, user_name, image_url, caption, created_at,
    from_correspondent (boolean — false = regular user, true = correspondent),
    gdrive_file_id (text — populated after Drive backup, nullable)

Migration 004_upload_slots.sql added the from_correspondent column and an index.
Run it in the Supabase SQL editor if it hasn't been applied yet:
  ALTER TABLE public.correspondent_posts
    ADD COLUMN IF NOT EXISTS from_correspondent boolean NOT NULL DEFAULT false;
  CREATE INDEX IF NOT EXISTS idx_posts_from_correspondent
      ON public.correspondent_posts (from_correspondent);
  ALTER TABLE public.correspondent_posts
    ADD COLUMN IF NOT EXISTS gdrive_file_id text;


DEPLOYMENT
----------
The app is deployed to Vercel from the main branch.

  1. Make code changes locally
  2. git add / git commit / git push origin main
  3. Vercel auto-deploys on push (or run: npx expo export --platform web)

The Supabase edge function is deployed separately:
  cd apps/tract-tracker
  supabase functions deploy upload-to-drive --no-verify-jwt

Supabase secrets are set with:
  supabase secrets set SECRET_NAME="value"
  (requires supabase link --project-ref vpgykxfbrfnojmgmzriq first)

SLOT POOL STATUS (HOW TO CHECK)
---------------------------------
In Supabase SQL editor:
  SELECT COUNT(*) FROM correspondent_posts WHERE from_correspondent = false;

If this number is >= 500, regular users will see "Upload slots are full" and
cannot upload. Correspondents are never affected by this count.

To raise or lower the cap, edit CorrespondentScreen.tsx:
  const REGULAR_UPLOAD_SLOTS = 500;   ← change this number
Then redeploy to Vercel.


MAKING SOMEONE A CORRESPONDENT OR ADMIN
-----------------------------------------
Option A — Database (permanent):
  UPDATE tract_users SET is_correspondent = true WHERE name = 'Their Name';
  UPDATE tract_users SET is_admin = true WHERE name = 'Their Name';

Option B — Hardcode in AuthContext.tsx (for email-based, no DB change needed):
  const ADMIN_EMAILS = new Set(['njcarlo@gmail.com', 'cogtv@gmail.com', 'newperson@example.com']);
  Then redeploy.
