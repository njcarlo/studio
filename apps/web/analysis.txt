1. OVERVIEW
--------------------------------------------------------------------------------

Name:        COG App (Church Operations and Governance)
Framework:   Next.js 15 (App Router) with Turbopack
Language:    TypeScript
Styling:     Tailwind CSS + Radix UI (shadcn/ui component library)
Font:        IBM Plex Sans (Google Fonts)

This is a church management web application for COG Dasmariñas. It handles
worker profiles, ministry management, room reservations, meal stubs, attendance
tracking, QR scanning, approvals, and C2S (Connect to Serve) discipleship groups.

It is part of a monorepo (Turborepo) alongside apps/tract-tracker and shared
packages (database, graphql, store, types, ui).

3. AUTHENTICATION
--------------------------------------------------------------------------------

The web app uses Firebase Auth (NOT Supabase Auth) for user login.

Flow:
  - FirebaseClientProvider (from @studio/database) listens to onAuthStateChange
  - Auth state is stored in Zustand via useAuthStore
  - UserRoleSyncerSQL component runs on every auth change:
      1. Fetches worker profile from Prisma DB by UID or email
      2. Fetches all roles and ministries
      3. Computes permission flags (isSuperAdmin, canManageWorkers, etc.)
      4. Syncs everything into usePermissionsStore (Zustand)
  - Password reset uses Firebase Admin to generate reset link, then sends via Resend

Auth is split:
  - apps/web  → Firebase Auth
  - apps/tract-tracker → Supabase Auth

This dual-auth setup is a known inconsistency. The .env has both Firebase
credentials (GOOGLE_APPLICATION_CREDENTIALS) and Supabase credentials.

6. KEY FEATURES
--------------------------------------------------------------------------------

A. Worker Management
   - Full CRUD via Prisma server actions
   - Paginated list with search + ministry filter
   - Worker stats (total, active, inactive, secondary)
   - Batch ministry assignment
   - QR token per worker for scanning

B. Approval Workflow
   - Types: New Worker, Ministry Change, Room Booking
   - Email notifications via Resend when new request submitted
   - Approvers resolved by ministry or role permissions
   - Side effects on approval: worker activated, ministry updated, booking status synced

C. Room Reservations
   - Calendar-based booking system
   - Conflict detection (implied by booking model)
   - Role-based approval flow
   - Venue elements (equipment) management

D. Meal Stubs
   - Manual and automatic generation
   - Stub types, assigned by named staff
   - QR scanner for validation at meal service

E. Attendance Tracking
   - QR-based clock in/out
   - Linked to worker profiles
   - Scan logs stored separately

F. ORS Sync (Legacy Migration)
   - Full sync from cogdasma.com/ors-reader/public API
   - Imports: workers, ministries, branches, areas, C2S groups, mentees, attendance
   - Worker diff view: shows new/updated/synced status per worker
   - Password hash migration: MD5, SHA1, SHA256, plain text → Firebase Auth
   - Firebase importUsers for batch hash migration
   - Paginated diff with field-level comparison

G. C2S (Connect to Serve)
   - Mentor groups with mentees
   - Synced from ORS legacy system
   - Analytics view (permission-gated)

H. AI Features (Genkit)
   - Google Genkit + Gemini integration
   - Dev mode only (genkit:dev script)
   - Not yet wired to production features (dev.ts only)

I. Email Notifications
   - Resend SDK for transactional email
   - Approval request notifications
   - Password reset emails
   - Falls back to console.log mock if RESEND_API_KEY not set

J. Impersonation
   - Super admins can view the app as any worker
   - useImpersonationStore (Zustand) tracks impersonated worker ID
   - UserRoleSyncerSQL fetches the impersonated worker's profile when active

9. ISSUES & CONCERNS
--------------------------------------------------------------------------------

CRITICAL:
  1. Dual auth systems — Firebase Auth for web, Supabase Auth for tract-tracker.
     This creates two separate user bases. Workers in the web app cannot log into
     tract-tracker with the same credentials without a separate account.

  2. TypeScript errors suppressed — next.config.ts has ignoreBuildErrors: true
     and ESLint is also ignored during builds. This masks real type errors.

  3. service-account.json committed to repo — Firebase Admin credentials file
     is referenced at ./service-account.json and appears to be in the repo root.
     This is a security risk if the repo is public.

  4. Credentials in .env — DATABASE_URL contains plaintext Supabase password.
     The .env file should never be committed to version control.

MODERATE:
  5. Debug endpoints exposed — /api/debug-prisma and /api/debug-workers are
     live API routes. These should be removed or protected before production.

  6. getWorkerStats uses roleId: { contains: 'secondary' } for secondary count
     which is a fragile string match, not a proper enum/flag check.

  7. AI features (Genkit) are only wired in dev mode (src/ai/dev.ts). No
     production AI flows are connected yet.

  8. Email sender is 'onboarding@resend.dev' (Resend sandbox). This needs to
     be changed to a verified domain sender for production.

  9. ORS sync fetches workers one-by-one in some paths (importOrsNewWorkers
     fetches each worker individually). For large imports this will be slow.

MINOR:
  10. build_output*.txt, dev_logs*.txt, errors.txt, tsc_errors.txt, etc. are
      leftover debug files in the repo root of apps/web. Should be cleaned up
      or added to .gitignore.

  11. users_temp.json and initial-data.json are data files sitting in the
      app root — likely leftover from seeding/migration work.

  12. The Capacitor mobile build (apps/web/android) appears to be a separate
      mobile wrapper around the web app. It has its own google-services.json
      and release keystore (cogapp-release.jks) committed to the repo.

  13. GraphQL layer (@studio/graphql) exists but it's unclear how much of the
      app uses it vs. direct Prisma server actions. Most data fetching appears
      to go through server actions in src/actions/db.ts.

  3. Move service-account.json out of the repo, use environment variable with
